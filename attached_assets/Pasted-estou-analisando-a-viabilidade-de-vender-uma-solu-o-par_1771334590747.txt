estou analisando a viabilidade de vender uma solução para o estado do Goias, que analise diferentes contratos de TI, de diferentes empresas, diferentes padrões de desenvolvimento, a principio pensei em uma solução que pegasse o o que foi desenvolvido naquele ciclo, analisasse e respondesse ao servidor publico resposavel pela aceitação dos pontos de função, como posso fazer isso?
Sua ideia é viável e está bem alinhada com o que órgãos públicos (incluindo estados) vêm adotando em contratações de TI baseadas em pontos de função (APF). Para estruturar essa solução, vale pensar em dois eixos: um motor de análise do software/contrato e um fluxo de resposta para o servidor público responsável pela aceitação.

1. Entender o modelo de contratação de TI no Goiás
Antes de codar, vale validar se o estado adota:

APF tradicional ou APF simplificado (SFP), conforme o roteiro de métricas do SISP, que já é usado como referência por vários entes públicos.
​

Outras métricas (como horas de serviço técnico) para contratos de TI.
​

Se o Goiás ou o órgão destino segue o padrão federal, o seu sistema pode ser desenhado em cima do SFP (mais simples e ágil) ou APF clássico, com documentação e critérios padronizados.

2. Saída do ciclo de desenvolvimento: o que a solução deve analisar
Você mencionou que a solução vai pegar “o que foi desenvolvido naquele ciclo”. Do ponto de vista de análise para pontos de função, o que importa, em geral, é:

Documentação funcional (especificação de requisitos, stories, protótipos, documentos de usuário).

Código fonte (se possível, para mapear CRUDs, interfaces, arquivos lógicos, etc.).

Artefatos de entrega (relatórios de testes, evidências de homologação, artefatos de sustentação).
​

Seu sistema pode:

Extrair automaticamente funcionalidades a partir de:

Requisitos em texto (IA/NLP para identificar transações, entradas, saídas, arquivos lógicos internos e externos).

Estrutura de código (endpoints de API, telas, entidades, bancos de dados).

Validar essas funcionalidades contra o padrão de contagem de pontos de função (NESMA/IFPUG).

3. Modelo da solução: como ligar análise ao servidor público
Você quer transformar esse ciclo em um sistema que responde ao servidor responsável. Um modelo prático é:

Passo 1 – Entrada do ciclo de entrega

O gestor ou fornecedor carrega:

Documentação (Word, PDF, Jira, etc.).

Código fonte (repositório ou ZIP).

Relatórios de entrega / testes.

Passo 2 – Análise automática

O sistema:

Faz análise de texto (NLP) para extrair funcionalidades.

Faz análise de código (AST, scanners de arquitetura) para identificar entidades, interfaces, endpoints, arquivos lógicos, etc.

Gera:

Tamanho funcional (APF/SFP) por ciclo.

Relatório de discrepâncias (funcionalidades prometidas vs. entregues, diferenças de escopo, funções não contabilizadas).

Passo 3 – Resposta ao servidor público
Aqui é o “pulo do gato” para o uso público:

O sistema gera um relatório de aceitação com:

Número de pontos de função contados (justificado por item).

Evidências de onde essas funções foram encontradas (trechos de documento, telas, endpoints).

Apontamento de:

Itens não entregues.

Itens fora de escopo.

Riscos de qualidade (ex.: falta de testes, inconsistência entre documentos e código).

O servidor pode:

Aceitar “apertando um botão” com base nesse relatório.

Solicitar revisão ao fornecedor, com lista de itens pendentes.

Gerar contra‑relatório (se sua solução for extensível a um sistema de gestão de contratos).

4. Como você pode estruturar tecnicamente
Para viabilizar essa solução, pense em:

Motor de análise:

NLP para análise de documentos: extrair funcionalidades, classificar em ALI, AIE, EE, SE, CE (seguindo o padrão de APF).

Análise de código: ferramenta customizada (ou integrada a ferramentas como SonarQube, CodeScene, etc.) para mapear estruturas de dados e transações.

Motor de pontos de função: tabela de regras e pesos (NESMA/IFPUG) para calcular o tamanho funcional.

Interface para o servidor público:

Dashboard simples com:

Tamanho funcional entregue vs. contratado.

Diferenciais por fornecedor (útil para comparar contratos de TI diferentes).

Relatórios exportáveis (PDF/ODT) para anexar ao processo administrativo.

Integração com processo de contratação:

Se o estado usa SISP ou roteiro de métricas de software, o sistema pode exportar o resultado em formato compatível com o padrão de demonstrativo de pontos de função.

2. Pilares da solução
Você pode dividir o pitch em 3 pilares:

Pilar 1 – Entrada dos dados

Documentos estruturados no processo:

Contrato e edital (PDF, ODT, Word, processo eletrônico).

Especificação de requisitos (stories, backlogs, documentos funcionais).
​

Repositório de código (GitHub, GitLab, Bitbucket, ZIP, etc.).

Pilar 2 – Interpretador inteligente (IA + regras)

Uma API de IA especializada treinada em:

Roteiro de Métricas do SISP (v2.3 / 3.0) e Guia de Contagem de Pontos de Função.

Regras de APF/SFP (funções de dados, transacionais, regras de contagem, peso, fator de impacto, etc.).

A IA:

Lê e interpreta o contrato, edital e requisitos.

Relaciona o que foi contratado (escopo) com o que foi entregado (código).

Detecta:

Funcionalidades contratadas mas não entregues.

Funcionalidades a mais.

Pontos de função que não condizem com o padrão (ex: contagem inflada, inconsistência).

Pilar 3 – Saída: relatório de análise para o servidor público

Relatório de aceitação ou não aceitação da entrega, com:

Tamanho funcional entregue (APF/SFP) x contratado.

Indicadores de qualidade (testes, cobertura, consistência entre documento e código).

Sugestão de motivação para o servidor público “aceitar” ou “solicitar complementação” com base em critérios objetivos.
